Project: Lithic-Analysis-Platform (Dibble)
Source: v3.5 research prompt (generated)
======================================================================

CONTEXT:
Dibble v3 implements a COLMAP-based photogrammetry pipeline that
turns photos of lithic artefacts into 3D meshes. The meshes are
currently in COLMAP's arbitrary coordinate scale — the shape is
correct but absolute measurements (length, width, thickness in mm)
are not meaningful. v3.5 aims to add scale bar auto-detection so
the pipeline outputs real-world-unit meshes ready for measurement.

An existing "scale_bar_cm" field is already in PhotogrammetryConfig
(docs/superpowers/specs/2026-05-26-photogrammetry-pipeline-design.md,
section 5.3) but is documented as a future enhancement and currently
unused.

The typical lithic photography setup uses:
- A turntable with the artefact placed at centre
- A scale bar (typically a rigid ruler or printed scale, 1–10 cm
  long, often with alternating black/white bands) placed next to
  the artefact
- 12–60 photos taken from varying angles around the turntable
- A fixed or handheld camera (smartphone or DSLR)

We need the approach to be:
1. Fully automatic in "Default" mode (user says "here are photos",
   gets a correctly scaled mesh, no manual steps)
2. Provide feedback if scale detection fails
3. Work with existing photo sets (not require special markers or
   printed targets placed in advance, though we can document best
   practices for placement)

======================================================================

DEEP RESEARCH PROMPT:

**Scale bar auto-detection for close-range photogrammetry of small
artefacts**: I need a detailed technical analysis of approaches to
automatically detect a scale reference in a set of close-range
photogrammetry photos and use it to rescale the resulting 3D mesh to
real-world units (millimetres). The scale reference is a small ruler
or printed scale bar (1–10 cm) placed next to the artefact during
photography.

Please research and report on:

## 1. Approaches

For each approach, describe:
- How it works at a technical level
- What computer vision / image processing techniques it uses
- Typical accuracy achievable (as % of true scale)
- Computational cost (seconds vs minutes)
- Sensitivity to: lighting conditions, background clutter, scale bar
  orientation, camera distance, lens distortion
- Whether it can work with existing photo sets (retrospective)
- Whether it requires calibration or prior knowledge of scale bar
  dimensions
- Implementation complexity (hours/days estimate for an experienced
  Python developer)
- Existing open-source implementations or libraries that could be
  leveraged

### Approach A: Known physical dimensions of an object in the scene
The user places a ruler or known-length object next to the artefact.
The algorithm locates the object in the photos (or in the sparse
point cloud), measures its extent, and computes the scale factor.
Sub-approaches:
  A1. Locate the scale bar in the sparse COLMAP point cloud (points
      lying on the scale bar surface) and measure its length from
      3D points
  A2. Locate the scale bar in individual 2D photos via object
      detection, then triangulate its length via COLMAP's known
      camera poses
  A3. Detect the scale bar's tick marks or gradients in 2D and map
      known physical spacing to pixel distances

### Approach B: Fiducial markers
Place fiducial markers (ArUco, AprilTag, checkerboard) of known size
in the scene. Detect them automatically and compute scale from the
known marker dimensions.
  B1. Printed ArUco/AprilTag on paper, placed next to artefact
  B2. Checkerboard calibration pattern with known square size
  B3. Custom lithic-specific marker (e.g., printed ring with known
      inner diameter)

### Approach C: Camera calibration from known sensor + lens
If the camera model and sensor size are known (from EXIF data or user
input), the lens distortion and focal length from COLMAP's self-
calibration can theoretically be converted to metric scale.
  C1. Use COLMAP's known camera parameters + sensor width from EXIF
      to compute absolute scale
  C2. Requires the user to specify camera sensor size (or auto-detect
      from camera model database)

### Approach D: Multi-view triangulation of a reference distance
The user takes two photos with a known camera baseline (e.g., camera
moved exactly 10 cm between two shots). The known baseline provides
the scale reference.
  D1. Requires the user to measure the camera displacement
  D2. Could be automated with a calibrated turntable + camera arm

## 2. Integration with COLMAP

COLMAP produces:
- A sparse point cloud (from the mapper stage) in PLY/TXT/BIN format
  with 3D points and their per-view visibility
- Camera poses (rotation + translation) for each input photo
- Intrinsic camera parameters (focal length, principal point,
  distortion coefficients) — but in arbitrary scale

Specific questions:
- Can COLMAP's sparse point cloud be used directly for scale bar
  detection? The points are in arbitrary units — but relative
  distances are correct. If we can identify two points on the scale
  bar that are 5 cm apart in reality, can we compute the scale
  factor from their distance in arbitrary units?
- Can we add constraints to COLMAP's bundle adjustment to fix the
  scale? COLMAP supports known camera positions (GPS) — could we
  inject a known distance constraint?
- What is the accuracy of relative distances in COLMAP's sparse
  reconstruction for small objects (10–50 mm scale)?
- Is the dense MVS point cloud more or less suitable than the sparse
  cloud for scale bar detection?

## 3. Practical considerations for lithic artefacts

- Scale bars are typically black-and-white rulers, 1–10 cm long
- The artefact is usually grey/brown stone — good visual contrast
  with a white/black scale bar
- Photos are taken against a plain background (turntable surface)
- Photo resolution: typically 12–48 MP (smartphone or DSLR)
- What scale bar designs maximise detectability? (Colour contrast,
  pattern, size relative to artefact)
- What is the minimum scale bar resolution in pixels needed for
  reliable detection? (e.g., "scale bar should occupy at least 50
  pixels of its narrowest dimension")
- How should the scale bar be positioned? (Flat on the surface?
  Vertical? Next to vs touching the artefact?)
- Can a single scale bar visible in a subset of photos be sufficient,
  or does it need to appear in all/most photos?

## 4. Existing tools and libraries

- OpenCV ArUco module (cv2.aruco) — mature, fast, well-documented
- AprilTag (AprilRobotics/apriltag) — robust, used in robotics
- COLMAP's point cloud processing tools
- Any existing photogrammetry software that does automatic scaling
  (Agisoft Metashape's scale bars, RealityCapture's known distance)
- Python libraries for point cloud processing (Open3D, Pyntcloud)

## 5. Fallback and user feedback

What should happen when scale detection fails?
- Fall back to unscaled mesh with a clear warning
- Allow manual scale input (user measures a known distance on the
  mesh and enters it)
- Show the user the detected scale candidates and let them confirm

## 6. Accuracy requirements

For lithic measurement, what accuracy is needed?
- Published inter-analyst caliper measurement variance for lithic
  length/width/thickness is typically ±0.5–2 mm
- What scale detection accuracy would make the photogrammetry
  pipeline competitive with hand measurements?
- How does scale accuracy affect downstream measurements (edge
  angles, platform angles — which are scale-invariant vs scale-
  dependent)?
